● MQTT SIMULATOR

DrValue 웨어러블
디바이스 시뮬레이터

웨어러블 기기의 생체 데이터를 시뮬레이션하여 MQTT로 발행하는 테스트 서버입니다. REST API로 디바이스를 등록하면, 현실적인 생체 데이터가 주기적으로 브로커에 발행됩니다.

[REST API] ──register──▶ [Publisher Server] ──publish──▶ [Mosquitto Broker] ──deliver──▶ [Subscriber] Express Node.js + mqtt.js Docker Java + Paho :3000 :1883

1 서버 실행 방법

Docker, Node.js, Java 환경이 필요합니다. 아래 순서대로 실행하세요.

MQTT 브로커 실행 (Mosquitto)
프로젝트 루트에서 Docker Compose로 Mosquitto 브로커를 실행합니다.
bash# 프로젝트 루트 디렉토리에서
docker-compose up -d

# 실행 확인
docker ps --filter name=mqtt-broker
✦ 브로커는 1883 포트에서 MQTT 프로토콜을 수신합니다. 이미 해당 포트를 사용 중이면 docker-compose.yml에서 포트를 변경하세요.
Publisher 서버 실행 (Node.js)
Express REST API 서버를 실행합니다. 디바이스 등록/관리와 MQTT 발행을 담당합니다.
bashcd publisher
npm install
npm start

# 출력:
# [Server] REST API 실행: http://localhost:3000
# [Server] 프리셋 테넌트: tenant-1, tenant-2, tenant-3, tenant-4, tenant-5
# [MQTT] 브로커 연결 완료: mqtt://localhost:1883
Subscriber 실행 (Java)
테넌트별로 MQTT 메시지를 구독합니다. 실행 인자로 테넌트 ID를 지정합니다.
bashcd subscriber
./gradlew run --args="tenant-1"

# 출력:
# [Subscriber] 브로커 연결 완료: tcp://localhost:1883
# [Subscriber] 테넌트: tenant-1
# [Subscriber] 토픽 구독: drvalue/tenant-1
# [Subscriber] 메시지 대기 중...
✦ 인자 없이 실행하면 기본값 tenant-1로 구독합니다. 다른 테넌트를 보려면 --args="tenant-2"처럼 변경하세요.

2 사용 방법

REST API로 디바이스를 등록하면 자동으로 MQTT 발행이 시작됩니다.

디바이스 등록

bashcurl -X POST http://localhost:3000/api/tenants/tenant-1/devices \
  -H "Content-Type: application/json" \
  -d '{
    "deviceId": "DEV-001",
    "modelName": "WF100",
    "intervalMs": 5000
  }'
응답 201{
  "deviceId": "DEV-001",
  "tenantId": "tenant-1",
  "modelName": "WF100",
  "intervalMs": 5000,
  "topic": "drvalue/tenant-1",
  "status": "publishing",
  "registeredAt": "2026-05-07T14:20:58.981Z"
}
✦ 등록 즉시 poweron 메시지가 1회 발행되고, 이후 intervalMs 간격으로 report가 발행됩니다. emergency는 매초 1~2% 확률로 발생합니다.

디바이스 목록 조회

bashcurl http://localhost:3000/api/tenants/tenant-1/devices

디바이스 제거 (발행 중지)

bashcurl -X DELETE http://localhost:3000/api/tenants/tenant-1/devices/DEV-001

REST API 전체 목록

MethodPath설명
GET /api/tenants 전체 테넌트 현황
GET /api/tenants/:tenantId/devices 디바이스 목록 조회
GET /api/tenants/:tenantId/devices/:deviceId 디바이스 상세 조회
POST /api/tenants/:tenantId/devices 디바이스 등록 → MQTT 발행 시작
DELETE /api/tenants/:tenantId/devices/:deviceId 디바이스 제거 → MQTT 발행 중지
! 프리셋 테넌트는 tenant-1 ~ tenant-5입니다. 존재하지 않는 테넌트에 요청하면 404를 반환합니다.

3 MQTT 토픽 구조

테넌트 단위 단일 토픽입니다. 하나만 구독하면 해당 테넌트의 모든 디바이스, 모든 메시지 타입을 수신합니다.

drvalue/{tenantId} ← 단일 토픽 (report, poweron, emergency 모두 여기로)
✦ drvalue/tenant-1 하나만 구독하면 해당 테넌트의 모든 디바이스 데이터 + 응급 이벤트까지 전부 수신됩니다. 메시지 내부의 "type" 필드로 report / poweron / emergency를 구분하세요.

구독 예시

토픽설명
drvalue/tenant-1 tenant-1의 모든 디바이스, 모든 메시지 타입 (report + poweron + emergency)
drvalue/+ 모든 테넌트의 모든 메시지 (와일드카드)

4 Payload 구조

모든 메시지는 JSON 형식이며, QoS 1로 발행됩니다. "type" 필드로 메시지 종류를 구분합니다.

📊

type: "report"

주기적 발행 
{
  "type": "report",
  "deviceId": "DEV-001",
  "tenantId": "tenant-1",
  "modelName": "WF100",
  "battery": 85.3,              // 배터리 (%), 100→0 점진 감소
  "breathRate": 18,             // 분당 호흡수
  "stepCount": 1234,           // 걸음수 (누적)
  "lmaCount": 45,              // 허리 움직임 횟수 (누적)
  "torsoAngleMean": 15.2,       // 허리 평균 각도 (°)
  "torsoAngleMin": 10.5,        // 허리 최소 각도 (°)
  "bentDuration": 5,            // 숙인 시간 (초)

  "gpsLatitude": 37.5665,       // 위도
  "gpsLongitude": 126.978,      // 경도
  "usageTimeSeconds": 3600,    // 사용시간 (초, 누적)

}

5 응급 이벤트

매초 1~2% 확률로 랜덤 발생합니다. type: "emergency" 메시지의 세부 타입은 다음과 같습니다.

emergencyTypeemergencyLabel설명
0낙상급격한 가속도 변화로 낙상 감지
1과호흡분당 호흡수가 비정상적으로 높음
2과활동과도한 신체 활동 감지
3과작업허리 과부하 작업 (장시간 숙임)

6 시뮬레이션 데이터 범위

각 필드는 현실적인 범위 내에서 시뮬레이션됩니다.

필드범위변화 패턴
battery 100 → 0 매 발행마다 0.1~0.5% 감소
breathRate 12~25 회/분 정규분포 (평균 17, 표준편차 3)
stepCount 0~50 / 주기 매 주기 누적 증가
lmaCount 0~10 / 주기 매 주기 누적 증가
torsoAngleMean 5~40° 매 주기 랜덤
torsoAngleMin 0~50° torsoAngleMean 기준 -5° 이내
bentDuration 0~120초 매 주기 랜덤
calories 누적 0.2×lma + 0.06×step
GPS 서울 기준 고정 좌표 ± 0.001
emergency 매초 1~2% 랜덤 발생, 4가지 타입
